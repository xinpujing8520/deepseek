DeepSeek API 的日志如果你是发 HTTP 请求调 DeepSeek API，那错误信息一般都会以 HTTP 状态码和 JSON 响应体给你返回。你程序（不管是Python、JS还是别的语言）抓到这些异常时，**务必**把完整的状态码和响应体都打出来。```python# 你的 DeepSeek API 调用代码response = client.chat.completions.create(model="deepseek-chat",messages=[{"role": "user", "content": "Hello DeepSeek!"}])print(response.choices[0].message.content)except Exception as e:print(f"DeepSeek API 调用失败：{e}")if hasattr(e, 'response') and e.response is not None:print(f"HTTP 状态码：{e.response.status_code}")print(f"错误详情：{e.response.text}")```很多API库会自动帮你把这些错误包起来，所以你可能得了解下它咋处理异常的。另外，DeepSeek 官网平台一般也会有 API 调用记录和错误日志，去用户后台瞅瞅，排查 API 问题贼好用。对[第一次 DeepSeek API 请求](/jiqiao/deepseek-api-first-request/)不熟的，赶紧看教程补补课！### 2. 本地部署模型的日志 (Ollama 等)如果你通过 Ollama 或其他工具在本地运行 DeepSeek 模型，日志通常会在以下地方：*   **终端/控制台输出：** 当你启动 Ollama 或运行模型时，所有的日志信息都会实时打印在你的终端窗口。这是最直接的查看方式。*   **Ollama 日志文件：** Ollama 通常会在其安装目录下生成日志文件。具体位置可能因操作系统而异（例如 Linux 上的 `~/.ollama/logs`）。*   **系统日志：** 在某些情况下，特别是涉及硬件或系统级别的错误时，你可能需要检查操作系统的系统日志（如 Linux 上的 `journalctl` 或 Windows 上的事件查看器）。总之，不管哪种方法，重点是把你程序或部署工具的日志级别开高点，把所有能抓到的错误信息都逮住！## 第三步：错误解读与问题排查找到“案发现场”和详细日志了，接下来咱就得像个侦探，好好分析这些信息，一步步挖出问题的根儿。### HTTP 状态码是好帮手遇到 API 错误，HTTP 状态码就是个好向导：*   `2xx`：成功，但可能包含警告信息。*   `400 Bad Request`：请求参数有问题。仔细检查你的请求体 JSON 格式、参数名是否拼写错误、参数值是否符合 API 规范。*   `401 Unauthorized`：认证失败。检查你的 API Key 是否正确、是否已过期，或者是否未在请求头中正确携带。*   `403 Forbidden`：权限不足。你的 API Key 可能没有访问该资源的权限。*   `404 Not Found`：请求的资源不存在。检查你的 API 端点 URL 是否正确。*   `429 Too Many Requests`：速率限制。你需要降低请求频率，或增加请求之间的延迟。*   `500 Internal Server Error`：DeepSeek 服务器内部错误。这通常是 DeepSeek 端的问题，你可以稍后重试，或查看 DeepSeek 官方状态页。*   `502 Bad Gateway`, `503 Service Unavailable`, `504 Gateway Timeout`：服务器或网络代理问题，也可能是 DeepSeek 服务暂时不可用。### 模型返回的具体错误信息除了状态码，DeepSeek API 通常会在 JSON 响应体中返回更详细的错误对象，例如：```json{"error": {"code": "invalid_api_key","message": "Incorrect API key provided: sk-**************.
